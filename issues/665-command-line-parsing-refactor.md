# 665-command-line-parsing-refactor. Formalize command line parsing with command maps

**Priority:** P3
**Status:** open

## Problem

Command line parsing in `fjs` uses `switch` statements that mix dispatch logic
with command metadata inline:

- `fs/fjs/module.f.ts` — top-level commands (`test`/`t`, `compile`/`c`, `cas`/`s`, `run`/`r`)
- `fs/cas/module.f.ts` — CAS subcommands (`add`, `get`, `list`)

Aliases, error messages, and handlers are scattered across `case` blocks with
no structured relationship between them. Adding a command requires a new `case`
block, and there is no way to enumerate available commands at runtime (so `help`
cannot be generated).

## Proposal

Replace each `switch` with a `Commands` list where each entry is self-descriptive
— it carries its own names (canonical + aliases):

```ts
type Command = {
  readonly names: readonly string[]   // names[0] is canonical; rest are aliases
  readonly description: string
  readonly handler: (args: readonly string[]) => Effect<NodeOp, number>
}

type Commands = readonly Command[]
```

A generic `dispatch(commands, args)` helper builds a lookup map from all `names`,
looks up `args[0]`, calls the matched handler with the remaining args, and
produces a consistent error message listing available commands when the command
is missing or unknown.

The `handler` type is an effect function for the first iteration. In the future
it may also accept a nested `Commands` to support both recursive subcommand
dispatch and auto-generated hierarchical help output.

Because descriptions live in the list, `fjs help` / `fjs --help` can be
auto-generated by iterating it — no duplicated documentation.

- [ ] Replace the CAS `switch` in `fs/cas/module.f.ts` with a `Commands` list
