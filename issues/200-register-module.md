# 200. `registerModule`: test registration for external frameworks

## Background

`runModule` in `fs/dev/tf/module.f.ts` drives the Effects-based `fjs t` runner.
It executes tests immediately, sandboxes each leaf function call individually,
accumulates `TestState` (pass/fail/time), and walks return-value sub-trees as
follow-on effects.

When bridging to an external framework (Node `--test`, Bun, Playwright), the
tests must be *registered* upfront at import time â€” the framework owns
scheduling and execution. `scanModule` in `fs/dev/tf/module.ts` currently
handles this with its own independent walk, duplicating traversal logic
([i155](./155-test-runner-integration.md)).

The goal of this issue is to replace that duplication with a `registerModule`
function that lives in `module.f.ts` and returns an `Effect`. `module.ts` then
becomes a thin shell that only creates the operation map and runs the effect:

```ts
// module.ts â€” only this remains
const map = { ... } // framework-specific operation handlers
await asyncRun(map)(registerModule(moduleMap))
```

All traversal logic stays in `module.f.ts`.

## Sandboxing scope difference

This is the core reason `runModule` cannot be reused directly.

In `runModule`, `Reporter.test` sandboxes only the leaf function:

```ts
// sandbox wraps only fn()
sandbox(fn) â†’ SandboxResult<unknown>
// sub-tree walk happens outside the sandbox, on the returned value r
walk([...testPath, null], false, r)
```

In `registerModule`, the test body registered with the framework must cover
the entire execution: calling `fn`, inspecting the return value, and running or
registering any sub-tests it contains. All of that happens *inside* the
framework's test callback. The sandbox must therefore wrap the whole unit.

## `Register<O>` interface

```ts
export type Register<O extends Operation> = {
    readonly register: (name: string, expectFailure: boolean, test: () => Effect<O, void>) => void
}
```

`register` returns `void` â€” it is the **operation handler** that the async
runner calls. The framework interaction (calling `nodeTest.test`, `t.test`,
etc.) lives entirely inside this handler.

`test` is a thunk that, when invoked by the framework's test callback, produces
`Effect<O, void>`. Because `O` includes the `RegisterTest` operation,
calling the `registerTest` Func inside the thunk emits further register effects
â€” this is how sub-tests generated by a function's return value are registered
recursively without any logic in `module.ts`.

## `RegisterTest` operation

`Register<O>` is the shape of the operation map entry for a `RegisterTest`
operation (already sketched in `module.ts`):

```ts
type RegisterTest =
    readonly['registerTest', (name: string, expectFailure: boolean, test: () => Effect<RegisterTest, void>) => void]

const registerTest: Func<RegisterTest> = do_('registerTest')
```

`registerModule` uses `registerTest(...)` to emit effects; the async runner
dispatches them to the concrete `register` handler in the operation map.

## `registerModule` design

```ts
const registerModule =
    (k: string, v: unknown): Effect<RegisterTest | All, void> => {
        const effects = collectTests([], false, v).map(
            ([path, set]): Effect<RegisterTest, void> =>
                registerTest(
                    fmtImport(k, path),
                    set.throws,
                    () => {
                        const [s, r] = tryCatch(set.fn)
                        if (!set.throws && s === 'ok') {
                            return all(...collectTests([...path, null], false, r).map(
                                ([subPath, subEntry]) =>
                                    registerTest(fmtImport(k, subPath), subEntry.throws, () => pure(undefined))
                            )).step(() => pure(undefined))
                        }
                        return pure(undefined)
                    }
                )
        )
        return all(...effects).step(() => pure(undefined))
    }
```

No `TestState` accumulation â€” the framework owns pass/fail counting.

## Framework-specific operation maps in `module.ts`

### Node `--test` with nested `t.test`

```ts
const subMap = (t: nodeTest.TestContext): ToAsyncOperationMap<RegisterTest> => ({
    registerTest: (name, expectFailure, test) =>
        t.test(name, { expectFailure }, () => asyncRun(subMap(t))(test())),
})

const rootMap: ToAsyncOperationMap<RegisterTest> = {
    registerTest: (name, expectFailure, test) =>
        nodeTest.test(name, { expectFailure }, async t =>
            asyncRun(subMap(t))(test())
        ),
}
```

### Bun / Playwright

The Bun map runs sub-tests inline inside the flat `fw.test` callback; the exact
shape may require adjustments to `Register<O>` once
[i183](./183-tf-framework-scenario-tests.md) scenario tests make the behaviour
observable.

## Relation to existing code

- `fs/dev/tf/module.ts` already has `RegisterTest`, `registerTest`, `map2`, and
  `run2`; those become the foundation for the Node operation map.
- `collectTests` (already exported) is reused as-is.
- `runModule` / `Reporter` are unchanged; this is an additive parallel path.
