# 200. `registerModule`: test registration for external frameworks

## Background

`runModule` in `fs/dev/tf/module.f.ts` drives the Effects-based `fjs t` runner.
It executes tests immediately, sandboxes each leaf function call individually,
accumulates `TestState` (pass/fail/time), and walks return-value sub-trees as
follow-on effects.

When bridging to an external framework (Node `--test`, Bun, Playwright), the
tests must be *registered* upfront at import time — the framework owns
scheduling and execution. `scanModule` in `fs/dev/tf/module.ts` currently
handles this with its own independent walk, duplicating traversal logic
([i155](./155-test-runner-integration.md)).

The goal of this issue is to replace that duplication with a `registerModule`
function that uses the same tree-walking infrastructure as `runModule` but
targets a `Register<O>` interface instead of `Reporter<O>`.

## Sandboxing scope difference

This is the core reason `runModule` cannot be reused directly.

In `runModule`, `Reporter.test` sandboxes only the leaf function:

```ts
// sandbox wraps only fn()
sandbox(fn) → SandboxResult<unknown>
// sub-tree walk happens outside the sandbox, on the returned value r
walk([...testPath, null], false, r)
```

In `registerModule`, the test body registered with the framework must cover
the entire execution: calling `fn`, inspecting the return value, and running or
registering any sub-tests it contains. All of that happens *inside* the
framework's test callback. The sandbox must therefore wrap the whole unit:

```ts
// framework invokes this at test execution time
nodeTest.test(name, async () => {
    // sandbox must cover fn() AND sub-tree discovery
    const [s, r] = tryCatch(fn)
    ...discover and run/register sub-tests from r...
})
```

If only `fn()` were sandboxed (as in `runModule`), an exception thrown during
sub-tree discovery would propagate uncaught into the framework rather than being
handled as a test failure.

## `Register<O>` interface

```ts
export type Register<O extends Operation> = {
    readonly register: (file: string, path: Path, set: TestEntry) => Effect<O, void>
}
```

`register` is called once per leaf test (same `TestEntry` as `Reporter.test`).
Its implementation decides:
- how to register the test with the specific framework
- what scope the sandbox covers
- how to handle sub-tests generated by the function's return value

The interface is intentionally minimal so it can be adjusted per framework as
behaviour differences are discovered (see [i183](./183-tf-framework-scenario-tests.md)).

## `registerModule` design

```ts
const registerModule =
    <O extends Operation>({ register }: Register<O>) =>
    (k: string, v: unknown): Effect<O | All, void> => {
        const effects = collectTests([], false, v).map(
            ([testPath, set]): Effect<O, void> =>
                register(k, testPath, set)
        )
        return all(...effects).step(() => pure(undefined))
    }
```

Because the framework owns pass/fail accounting, there is no `TestState`
to accumulate. The only job of `registerModule` is to fan out `register`
calls for every statically reachable leaf test in the module.

## Framework-specific `Register` implementations

### Node `--test` (supports nested `t.test`)

```ts
const nodeRegister: Register<never> = {
    register: (file, path, { fn, throws }) => {
        nodeTest.test(fmtImport(file, path), async t => {
            const [s, r] = tryCatch(fn)
            if (throws === (s === 'ok')) { throw r }
            if (!throws && s === 'ok') {
                for (const [sub, entry] of collectTests([...path, null], false, r)) {
                    t.test(fmtImport(file, sub), () => runEntry(entry))
                }
            }
        })
        return pure(undefined)
    }
}
```

Node allows `t.test(...)` inside a running test callback, so sub-tests can be
registered dynamically after the function executes.

### Bun / Playwright (no nested registration)

Bun and Playwright do not support registering new tests from inside a running
test callback. Sub-tests must be run inline:

```ts
const bunRegister: Register<never> = {
    register: (file, path, { fn, throws }) => {
        fw.test(fmtImport(file, path), () => {
            const [s, r] = tryCatch(fn)
            if (throws === (s === 'ok')) { throw r }
            if (!throws && s === 'ok') {
                for (const [, entry] of collectTests([...path, null], false, r)) {
                    runEntryInline(entry)  // throws on failure
                }
            }
        })
        return pure(undefined)
    }
}
```

This inline execution is the existing behaviour of `scanModule` in `module.ts`;
`registerModule` + `bunRegister` would replace it. The known breakage of
return-value sub-trees on Bun ([i155 §3](./155-test-runner-integration.md),
[i125](./README.md)) is visible here: `fw.test(...)` is called at import time
with a static name; any sub-tests generated by `fn()` cannot be registered as
separate top-level tests.

## Relation to existing code

- `fs/dev/tf/module.ts` already sketches a `RegisterTest` effect and
  `registerTest` Func; those can become the operation backing the Node
  `Register` implementation once this design is in place.
- `collectTests` (already exported) is reused as-is — no additional exports
  needed from `module.f.ts`.
- `runModule` / `Reporter` are unchanged; this is an additive parallel path.
