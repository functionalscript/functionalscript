# 200. `registerModule`: test registration for external frameworks

## Background

`runModule` in `fs/dev/tf/module.f.ts` drives the Effects-based `fjs t` runner.
It executes tests immediately, sandboxes each leaf function call individually,
accumulates `TestState` (pass/fail/time), and walks return-value sub-trees as
follow-on effects.

When bridging to an external framework (Node `--test`, Bun, Playwright), the
tests must be *registered* upfront at import time — the framework owns
scheduling and execution. `scanModule` in `fs/dev/tf/module.ts` currently
handles this with its own independent walk, duplicating traversal logic
([i155](./155-test-runner-integration.md)).

The goal of this issue is to replace that duplication with a `registerModule`
function that uses the same tree-walking infrastructure as `runModule` but
targets a `Register<O>` interface instead of `Reporter<O>`.

## Sandboxing scope difference

This is the core reason `runModule` cannot be reused directly.

In `runModule`, `Reporter.test` sandboxes only the leaf function:

```ts
// sandbox wraps only fn()
sandbox(fn) → SandboxResult<unknown>
// sub-tree walk happens outside the sandbox, on the returned value r
walk([...testPath, null], false, r)
```

In `registerModule`, the test body registered with the framework must cover
the entire execution: calling `fn`, inspecting the return value, and running or
registering any sub-tests it contains. All of that happens *inside* the
framework's test callback. The sandbox must therefore wrap the whole unit.

## `Register<O>` interface

```ts
export type Register<O extends Operation> = {
    readonly register: (file: string, path: Path, set: TestEntry) => Effect<O, void>
}
```

`register` is called once per leaf test. It returns a *pure effect* — it does
not call the framework directly. The framework interaction is entirely delegated
to the operation map that runs the effect.

## Operation: `RegisterTest`

The `RegisterTest` operation (already sketched in `module.ts`) is the bridge:

```ts
type RegisterTest =
    readonly['registerTest', (name: string, expectFailure: boolean, test: () => Effect<RegisterTest, void>) => void]
```

The third argument is a *thunk* that, when called inside the framework's test
callback, produces an effect that registers whatever sub-tests were generated by
the test function's return value. The operation map decides how to execute it.

The `Register` implementation uses this operation:

```ts
const register =
    (file: string, path: Path, { fn, throws }: TestEntry): Effect<RegisterTest, void> =>
    registerTest(
        fmtImport(file, path),
        throws,
        () => {
            const [s, r] = tryCatch(fn)
            // sub-tests generated by fn's return value are registered
            // as nested registerTest effects inside this thunk
            if (!throws && s === 'ok') {
                return all(...collectTests([...path, null], false, r).map(
                    ([sub, entry]) => register(file, sub, entry)
                )).step(() => pure(undefined))
            }
            return pure(undefined)
        }
    )
```

`register` is fully pure — it only builds an effect tree. No framework API is
called here.

## Operation maps decide framework behaviour

### Node `--test` with nested `t.test`

The root map uses `nodeTest.test`; the sub-map (called inside the callback)
uses `t.test` so sub-tests appear as children in the reporter:

```ts
const subMap = (t: nodeTest.TestContext): ToAsyncOperationMap<RegisterTest> => ({
    registerTest: (name, expectFailure, test) =>
        t.test(name, { expectFailure }, () => asyncRun(subMap(t))(test())),
})

const rootMap: ToAsyncOperationMap<RegisterTest> = {
    registerTest: (name, expectFailure, test) =>
        nodeTest.test(name, { expectFailure }, async t =>
            asyncRun(subMap(t))(test())
        ),
}
```

### Bun / Playwright (flat, no nested registration)

Bun cannot register new tests from inside a running test callback. The map runs
the thunk but ignores the resulting sub-`registerTest` effects, executing sub-tests
inline instead. The exact shape of this map is TBD and may require adjustments
to the `Register` interface (see below).

## `registerModule` design

```ts
const registerModule =
    <O extends Operation>({ register }: Register<O>) =>
    (k: string, v: unknown): Effect<O | All, void> => {
        const effects = collectTests([], false, v).map(
            ([testPath, set]) => register(k, testPath, set)
        )
        return all(...effects).step(() => pure(undefined))
    }
```

No `TestState` accumulation — the framework owns pass/fail counting.

## Future interface adjustments

The `Register` interface is intentionally minimal. As framework-specific
behaviours are discovered (especially via the scenario tests in
[i183](./183-tf-framework-scenario-tests.md)), additional fields or operation
variants may be needed. The Bun/Playwright case is the most likely driver of
such changes.

## Relation to existing code

- `fs/dev/tf/module.ts` already has a `RegisterTest` type, `registerTest` Func,
  and `map2`/`run2` sketch; those are the foundation for the Node operation map.
- `collectTests` (already exported from `module.f.ts`) is reused as-is.
- `runModule` / `Reporter` are unchanged; this is an additive parallel path.
